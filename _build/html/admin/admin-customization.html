Customization¶

Status¶

Status show the stage of a Request, Problem, Change and an Asset in their respective process flow. Statuses are essential in managing the life-cycle of tickets and CIs in the Flotomate. A lot of automatic processes can be created based on the Status of a ticket.

We have predefined Statuses for each of the ticket types and Assets. Along with pre-defined Statuses, you can add custom Statuses.

Note

Setting Custom Status requires administrative rights.

Predefined Status¶

Flotomate has predefined Statuses which are immutable. They coexist with custom Statuses. They are as follows:

  • Request:

    Here Statuses show the life-cycle of a Request.

    1. Open
    2. Pending On Requestor
    3. Pending in Approval
    4. Pending On Technician
    5. Resolved
    6. Closed

  • Problem:

    Here Statuses show the life-cycle of a Request.

    1. Open
    2. Pending On Requestor
    3. Pending in Approval
    4. Pending On Technician
    5. Resolved
    6. Closed

  • Change:

    Here each status is part of a stage in a Change process.

    1. Submitted:

      1. Requested
      2. Rejected
      3. Accepted

    2. Planning:

      1. In Progress
      2. Canceled
      1. Completed

    3. Approval:

      1. Pending
      2. Rejected
      3. Approved

    4. Implementation:

      1. In Progress
      2. Rejected
      3. Completed

    5. In Review:

      1. In Progress
      2. Failed
      3. Completed

  • Asset:

    Here statuses reflect the state of an Asset.

    1. In Use
    2. Missing
    3. Retired

Add Custom Status¶

  • Go to Admin >> (Request, Problem, Change or Asset) Custom Status.
  • Select a Status Type.
  • Use the Plus Icon to add a custom Status. You can add multiple Statuses.

Rearrange Custom Status¶

  • Go to the Custom Status page.
  • You can change the order of custom Statuses in all type by drag and drop.

Business Services in Problem Management¶

Flotomate allows you to add a list of business deliverables (services) of your organization. You can describe each and every business service that you add.

In the product, a Technician can highlight the impact of a Problem by associating it with a Business Service.

Note

Managing Business Services requires administrative rights.

View Business Services

  • Go to Admin >> Business Services (Problem Management).
  • In the Business Services page, you can view all your existing business service descriptions, if any.

Add Business Services¶

  • Go to the Business Services page.
  • Click Add Business Service button situated in the top right corner of the page.
  • Fill the Name and Description fields and click on Add.
  • In the Business Services page, you can edit any business description using the Edit Icon or delete using the Delete icon.

Templates¶

Templates in Flotomate allows you to quickly fill the create form of Request, Problem, and Change. You can create templates for each of the ticket types, and your users can use them while creating a Request, Problem or Change.

Note

Managing templates requires administrative rights.

Add a Template¶

  • Go to Admin >> (Request, Problem or Change) Templates.

  • In the new page, you can view your existing templates. To add a new one, click on Create a (Request, Problem, or Change) Template in the top right corner.

  • A dialog box opens where you can pre-define the following things:

    1. Request:

      1. The subject of a Request.
      2. Status, Urgency, and Impact of a Request.
      3. Tags of a Request.
      4. Category of a Request.
      5. Description of a Request.

    2. Problem:

      1. Subject to a Problem
      2. Whether a Problem is known or not.
      3. Status, Urgency, Nature of Problem and Impact of a Problem.
      4. Tags of a Problem.
      5. Category of a Problem.
      6. Description of a Problem.

    3. Change:

      1. The subject of a Change.
      2. Status, Urgency, Risk, Change Type and Impact of a Change.
      3. Tags of a Change.
      4. Category of a Change.
      5. Description of a Change.

Edit Request Templates¶

  • Go to (Request, Problem or Change) Templates page
  • Click the Edit-Icon adjacent to a template to open a dialog box.
  • Make your changes and hit Update. You are taken back to the Templates page
  • Delete any template using the Delete Icon next to the Edit Icon.

Custom Fields¶

Custom fields are additional fields added by a user. They can appear while creating a ticket, and in the details view of a ticket and CI. Custom fields can be added to Requests, Problems, Changes, and Assets.

Note

Adding Custom Fields requires administrative rights.

Add a Custom Field¶

  • Go to Admin >> (Request, Problem, Change or Asset) Custom Fields.
  • In the Custom Fields page, you can view your existing custom fields for each module, if any.

  • Regardless of the module, you can only have two types of fields: Text Field and Selection Field. A Text field has a label and an input area for a value, and a Selection Field has a label and an input area with a drop-down list of values.

    Anywhere, you can have both Text-Field and Selection-Field. You start adding fields using the Add button.

    Text Field

    • Field Label is the name of the field and appears above the field.

    • Default Value always appears in the field before being edited.
    Selection Field

    • You can have one Field Label but multiple options.

    • Use the Plus Icon to add more options

    • The options are shown as a list.

    • You can select an option to be the default value. Default Value always appears in the field before being edited.

    • You can change the Default Values’.

  • The behavior and outlook of custom fields change across the module types.

    1. Request:

    1. You can show a custom field in Create a Request form.

      For example, we have created a field called Employee ID in Fig. 468.

    2. New fields appear in the Work tab of a Request.

    1. You can make a field compulsory for certain Statuses. For example, if a field is compulsory in Open Status, then you cannot keep the field empty when changing to another Status.
    1. Problem:
    1. All custom fields appear in the Custom Fields tab under
      Analysis.
    1. You can make a field compulsory for certain Statuses. For example, if a field is compulsory in Open Status, then you cannot keep the field empty when changing to another Status
    1. Change:
    1. You can add a field to a particular stage of a Change.
    1. You can mark a field as compulsory at a particular stage.
    2. Fields appear in a separate section called Custom Fields.
    1. Asset:
    1. You add a field either to an Asset Type or a particular sub-type.
    2. Within a type, you can either make the field appear in Property section or Component section. Please refer to the Asset Management manual to learn more.

    1. Requestor

      1. Here Custom fields allow you to create new variables to store information about requestors.
      2. You can create n number of text fields.
      1. Such fields appear on the form for creating Requestors.
      1. Custom fields appear while mapping field names during LDAP Configurations.
      1. Custom fields are considered when mapping columns during CSV import of Requestors.

Edit Custom Fields¶

  • Go to the Custom Fields page.
  • .You can change the name of a custom field using the Edit Icon or delete it using the Delete Icon.

Category¶

The category is one of many categorization features we have. You can use category for delegation, workflow design, or any other categorization function required by your organization. You can have Categories for Request, Problem, Change, Project, and Software.

Predefined categories¶

Flotomate comes pre-loaded with the following Categories:

Request
  • Service Request
  • Incident

You can change the name of the above categories but not the Category Key.
Problem
  • General
  • Hardware
  • Software
  • Network
  • IT Administration

You can change the name of the above Categories.
Change
  • General
  • Process
  • Hardware
  • Software
  • Network
  • Security

You can change the name of the above Categories.
Project
  • Business
  • Maintenance
  • Infrastructure
  • Research

You can change the name of the above Categories.
Software
  • A/C
  • DB
  • Development
  • Game
  • Graphics
  • Internet
  • Multi-Media
  • Operating System
  • Others.

You can modify or delete the above categories.

Add a Category¶

Note

Adding a category requires you to have access to the Admin section.

  • Go to Admin >> (Request, Problem, Change, Project or Software) Categories.
  • In Request, Problem, Change, and Project, you can add a Category and its Sub-Categories up to level three. In Software, you can have only standalone Categories.
  • In the leftmost column, you can add a level one Category using the Add button. Clicking on a Category activates its right side Sub-Category column where you can add another Category.
  • In Request, you are asked for a Category Key when adding a new Category. This is used as a prefix to the ID of a Request.

Edit Category¶

  • Go to Categories page.
  • You can change the name of a Category using the Edit icon.
  • You can only delete Categories in Software. In other types, you have to turn a Category off.

Custom Rules¶

Using Custom Rules, you can set pre-requisites that need to be done before performing certain actions. You can have Custom Rules for Request, Problem, Change, Knowledge and Remote Deployment.

To Open Custom Rules:

  • Go to Admin >> (Request, Problem, Change, Knowledge or Patch) Custom Rules.

Current Custom Rules are as follows:

Request¶

  • Before resolving a Request:
    1. User Interaction:
      1. Should have at least one message in Ask Requestor tab.
      2. A Request should have at least one note.
      3. Should have at least one solution.
    2. Mandatory fields:
      1. Should have an assigned Technician.
      2. Should have a Location.
      3. Should have an Estimated Time.
      4. Should have a Diagnosis.
      5. Should have a Source.
      6. Should have Approval status as either Approved or Pre-Approved.
    3. Should have all Tasks as closed.
  • Before closing a Request:
    1. Same as before resolving a Request.
    2. Should not skip Resolve status.
  • Required to add a Note before:
    1. Should add a Note to a Request before performing the following actions:
      1. Changing assigned Technician of a Request.
      2. Changing Support Level of a Request.
      3. Changing Due Date of a Request.
      4. Changing Category of a Request.
      5. Changing Department of a Request.
      6. Changing Priority of a Request.

Problem¶

  • Before resolving a Request:
    1. User Interaction:
      1. A Request should have at least one note.
      2. Should have at least one solution.
    2. Mandatory fields:
      1. Should have an assigned Technician.
      2. Should have a Location.
      3. Should have a Root Cause.
      4. Should have a Symptom.
      5. Should have an Impact.
      6. Should have Approval status as either Approved or Pre-Approved.
    3. Should have all Tasks as closed.
  • Before closing a Request:
    1. Same as before resolving a Request.
    2. Should not skip Resolve status.
  • Required to add a Note before:
    1. Should add a Note before performing the following actions:
      1. Changing assigned Technician.
      2. Changing Due Date.
      3. Changing Category.
      4. Changing Department.

Change¶

You can set rules for each stage of a change process.

  • Submit:
    1. Should have a Change Manager.
    2. Should have an assigned Technician.
    3. Should have a Location.
  • Planning:
    1. Should have schedule and rollout dates.
    2. Should have a rollout plan.
    3. Should have an assigned Technician.
    4. Should have set impact.
    5. Should have a Backout plan.
  • Implementation:
    1. Should have a change implementer.
    2. All task must be closed before moving to the next stage.
    3. Should have an assigned Technician.
  • In Review:
    1. Should have a change reviewer.
    2. Should have an assigned Technician.
  • Require Note before:
    1. Changing category.
    2. Changing assigned Technician.
    3. Changing Department.
    4. Changing Location.

Asset¶

  • Manual Update:

    Note

    Asset rules are applicable for both discovered and manually created Assets.

    1. Whether to allow Technicians to update Hardware Components of an Asset.
    2. Whether to allow Technicians to update Software Components of an Asset.
    3. Whether to allow Technicians to update Users details of an Asset.

Knowledge¶

Whether Knowledge should be approved by approvers before publishing.

Remote Deployment:¶

Whether a deployment of Patches and Packages needs to go through an Approval process.
